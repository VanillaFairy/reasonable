# T0.1 — Work-order status is a ledger fold

**Spec:** §5.1 (F2, F8). **Wave:** 0a. **Depends on:** nothing. **Model:** capable (this is the linchpin
every later status consumer depends on).

## Objective

Make work-order status a **fold of the ledger**, not a field read from `journal.workOrders`. Today
`reconcile()` derives WO status only from the journal (loop over `journal.workOrders`, ~lines 118-192), so
a WO that exists in `ledger.jsonl` + live worktrees but is missing from the journal is **invisible** — the
exact shape of the motivating incident. After this task the ledger is the source and the journal is a
cross-checked cache.

## Scope

- **Create:** `lib/wo-status.mjs`, `test/wo-status.test.mjs`.
- **Edit:** `lib/reconcile.mjs` (make its WO-status derivation consume the fold; cross-check journal).
- **Do NOT edit:** `agents/journal-writer.md` (that is T0.4 — the journal keeps writing `status` for now;
  you only stop *trusting* it as the source), `ledger.mjs`, `progress-map.mjs`, `docs/*`.

## Interface (from shared/interfaces.md — use exactly)

```js
export function foldWorkOrderStatuses(events)  // -> Map<string, WoState>
// WoState = { status:'pending'|'running'|'blocked'|'dropped'|'done', lastSeq, blockedBy?, droppedBy? }
```

Semantics are pinned in `shared/interfaces.md` §T0.1 — implement them exactly:
- `running` = last `node-dispatched` for the WO with no later terminal (`node-completed`/`node-failed`/
  `node-canceled`).
- `blocked` = last terminal is `node-failed` with no later `ratification`/`amendment` whose
  `resolvesSeq === <that node-failed seq>`.
- `dropped` = an `amendment` with `drops:[{workOrder:id}]`, no later restoring `ratification`.
- `done` = `node-completed`/terminal `merged`, no later reopening `node-dispatched`.
- `pending` = planned/dispatched but the above yields nothing later; a WO with no events is absent.
- **Ignore `next-action` events** if present (forward-compat with Layer 2).

Note on WO identity in events: lifecycle events address the node by `node`/`workOrder` (see
`ledger.mjs` `resolveFamily1Address`); a WO id in the fold is the base work-order node id (attempts are
`base[k]` siblings — a reopen returns the WO to `running`). Read `ledger.mjs` to see how attempts/addresses
resolve; your fold works at the base-WO granularity.

## Positive constraints (DO)

- Write `foldWorkOrderStatuses` as a **pure** function (no fs, no git) over an events array; sort a copy by
  `seq` internally so it is order-independent.
- In `reconcile()`, derive each WO's status from the fold as the **authoritative** value, then keep the
  existing git-aware crash-recovery decisions (a `running` WO with no worktree and 0 commits ahead still
  downgrades to pending via the existing `node-downgraded` append path — do not remove that).
- When `journal.workOrders[id].status` exists and disagrees with the fold, push a **note**
  (`WO <id>: journal status '<j>' disagrees with ledger fold '<f>' — using ledger`) and use the fold.
- Surface WOs present in the ledger fold but **absent from the journal** (the incident case) so they are no
  longer invisible — at minimum include them in the reconcile result / notes as `running`/`blocked` per the
  fold, rather than dropping them.

## Negative constraints (DO NOT)

- Do NOT invent a status vocabulary beyond the five pinned values.
- Do NOT change the `journal-writer` or stop the journal from writing `status` (T0.4 does that; this task
  must leave the suite green with the journal *still* writing status).
- Do NOT match resolution by scanning for coincidental work-order-id mentions — only `resolvesSeq`.
- Do NOT regress any existing reconcile test. Where a test asserted "journal status is the source of truth,"
  update it to assert "ledger fold is the source, journal is cross-checked," and explain why in your report.

## Tests (test-first)

Write `test/wo-status.test.mjs` first, covering at least:
- dispatched-no-terminal → `running`; dispatched→completed → `done`.
- failed-then-no-resolution → `blocked`; failed-then-`ratification{resolvesSeq}` → back to `pending`/`done`
  per what the ratification implies (pin: a resolving ratification clears `blocked`; if no later dispatch,
  it is `pending`).
- amendment `drops` → `dropped`; drop then restoring ratification → not dropped.
- reopen (`node-failed` then later `node-dispatched`) → `running`.
- **the incident shape:** a WO present in the ledger (dispatched, no terminal) but absent from
  `journal.workOrders` → fold reports `running` (a reconcile-level test or a direct-fold test that proves
  it is no longer invisible).
- order-independence (shuffle events → same result).

Then a focused reconcile test proving reconcile now uses the fold (journal disagrees → note + fold wins).

## Acceptance

- `node test/wo-status.test.mjs` green; whole suite green (`for t in test/*.test.mjs; do node "$t"; done`).
- `reconcile()` status derivation reads the fold; journal is cross-check-only; incident WO is visible.
- No new dependency; `lib/wo-status.mjs` is pure and node-builtin-only.

## Context

This is the foundation of Layer 0. T0.4 (retire the journal `status` field) and all of Layer 2's
next-action projection consume this fold. Get the semantics exactly right — a wrong `dropped`/`blocked`
rule here silently corrupts scheduling later. The reconcile result object shape is large (see
`reconcile.mjs` ~lines 386-419); thread the fold in without breaking its consumers (`terminalWorkOrders`,
`liveLanes`, `briefing`).
