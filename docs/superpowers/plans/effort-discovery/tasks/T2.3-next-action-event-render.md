# T2.3 — the `next-action` ledger event + mirror render + Windows rename hardening

**Spec:** §7.1 (persistence rides the ledger, not the mirror; mechanical staleness). **Wave:** 2c (serial
after T2.2). **Depends on:** T2.2 (`result.nextAction`), all Layer 0/1. **Model:** most-capable (the
regen-render interaction + the K-staleness counter are subtle correctness).

## Objective

Persist the T2.2 projection into the truth log as a **`next-action` ledger event**, and render the **latest**
such event into the progress mirror (`progress.json.nextAction` + a `▶ NEXT` block in `progress.md`) so the
directive **survives the wholesale mirror regen by construction** (§7.1). Fold in the deferred Windows
`renameSync` EPERM/EBUSY hardening (layer0-checkpoint flag #4) while `atomicWrite` is open. **Do NOT add the
self-check (T2.4)** — this task persists + renders the projection as-is; T2.4 inserts the adversarial gate in
front of the append.

## Scope
- **Edit:** `lib/ledger.mjs` (add the event type + validator), `lib/reconcile.mjs` (append the event; add a
  NEXT line to `briefing()`), `lib/progress-map.mjs` (render the latest event; `atomicWrite` retry), tests.
- **Do NOT edit:** `lib/next-action.mjs` (frozen from T2.2 — you consume `result.nextAction`, do not change
  `projectDirectives`), `lib/route.mjs`, docs (T2.doc). Do NOT add `selfCheckDirectives` or any guard/drop

## The deltas (verify each against the merged code before writing)

### 1. `lib/ledger.mjs` — the `next-action` event type (Family 3)
- Add to `EVENT_SCHEMAS`: `'next-action': { required: [], validate: validateNextAction }`. It is **Family 3**
  (the implicit "everything else" branch — do NOT add it to `FAMILY_1_TYPES`/`FAMILY_2_TYPES`).
- `validateNextAction(event)` — mirror the existing `validateDropsAndResolvesSeq` shape:
  - `directives`, when present, must be an **array**; each element an object with a **non-empty-string `kind`**
    (the other directive fields — `slice`/`workOrders`/`workOrder`/`detail` — are free-form, don't over-validate).
  - `computedFrom`, when present, a **positive integer** (a 1-based ledger seq).
  - Return `undefined` on success, `{ ok:false, error }` on violation (match the sibling validator's return).
- Event shape: `{ type:'next-action', directives:[…], computedFrom:<seq> }`.

### 2. `lib/reconcile.mjs` — append the event; surface it in the briefing
- **Append** (after `result.nextAction` is computed — near the end, before `return result`): re-read the
  ledger's **current latest seq** fresh (`readJsonl(ledgerPath)`; the in-scope `ledger` array is stale after
  reconcile's own `node-downgraded` appends), set `computedFrom = <that latest seq>` (0 if empty), and
  `append(effortRoot, { type:'next-action', directives: result.nextAction, computedFrom }, { regen:true })`.
  The `regen:true` is what triggers the mirror render (delta 3). On `!ok`, `notes.push(...)` — a mirror-only
  failure is advisory, never fatal to reconcile (fail-open).
- **reconcile now appends one `next-action` event PER CALL.** This is a deliberate behavior change (reconcile
  was write-only-on-crash-recovery before). Update any test that asserts an exact post-reconcile ledger length
  or "reconcile wrote nothing," with a spec-tied note (§7.1: the projection is a recorded event, like the
  verdicts the ledger already carries). The fold ALREADY ignores `next-action` for WO status (wo-status.mjs,
  forward-compat) so this never perturbs status.
- **`briefing()`** — add a NEXT line rendering `result.nextAction` (the single-effort path shows `briefing(r)`;
  the multi-effort path already reads `progress.json.nextAction`). Render the directive set compactly, e.g.
  `` NEXT: DISPATCH slice expr-eval → WO-12, WO-13 · RUNNING WO-11 `` — one line, beside the existing
  `Lifecycle:` line. Empty set → omit or `` NEXT: (idle) ``.

### 3. `lib/progress-map.mjs` — render the LATEST `next-action`, survive the regen, harden the rename
- **The mirror regenerates WHOLESALE on every append** (confirmed: `writeMirror` → `buildTree` replays the
  whole ledger). So the directive must be re-derived from the ledger on every regen — never poked into
  `progress.json` out of band. In `writeMirror`/`composeProgressMd`: read the ledger (`readJsonl`), find the
  **latest** `next-action` event, and render it into:
  - `progress.json.nextAction` — a **string** (session-start.mjs:121 reads `p.nextAction` as a trimmed
    string; that is the contract to match). Absent any `next-action` event → omit the key (or `null`);
    session-start already falls back.
  - a **`▶ NEXT` block** atop `progress.md` (slot it beside the existing `⚠ inbox` blockquote, after the
    header/counts line).
  - **Mechanical staleness (§7.1):** append `— computed at seq <computedFrom>, <K> event(s) since` where
    **`K` = the count of ledger events with `seq > computedFrom` whose `type !== 'next-action'`** (a
    `next-action` event is a projection, not a state change, so it never counts toward its own staleness;
    right after reconcile appends it, `K === 0` → render `— fresh` or `0 events since`). K grows as real
    events land, making "a directive with K>0 is a hint, not an order" mechanically checkable.
- **`EVENT_MAP`:** give `next-action` an entry that produces **no tree op** (it is a header projection, not a
  lifecycle node — return `[]`, so it neither creates a tree node nor falls to `legacyFallback`'s spurious note).
- **Windows rename hardening (layer0-checkpoint flag #4):** in `atomicWrite`, wrap the `renameSync` in a
  **bounded retry on `EPERM`/`EBUSY`** (a concurrent Windows reader can transiently block the rename; today it
  throws → swallowed as advisory `mirrorError` → mirror lags one behind). A few attempts with a tiny
  synchronous backoff, then rethrow as today (still advisory upstream, never fatal). Only retry on
  `EPERM`/`EBUSY` — every other error rethrows immediately.

## Test discipline (test-first + discriminator)
Write tests FIRST. The KEY discriminators (each RED on the pre-task commit):
- **regen-clobber regression (§9, the crux):** reconcile an effort → the `▶ NEXT` block + `progress.json.nextAction`
  are present; then `append(root, {type:'node-checkpointed', node:...}, {regen:true})` (a NON-next-action
  event) → the NEXT block is **still there, re-rendered** (never erased), and `K` has incremented by 1. This
  proves the directive survives the wholesale regen — the whole point of §7.1.
- **latest-next-action-wins:** two `next-action` events in the ledger → the mirror renders the LATEST only.
- **the K counter:** K counts only non-`next-action` events after `computedFrom`; a later `next-action` append
  does not inflate K.
- **ledger validation:** a `next-action` with a malformed `directives` (non-array, or an element missing
  `kind`) is rejected by `append`; a well-formed one is accepted; `computedFrom` non-positive-int rejected.
- **the render carries the directives:** a DISPATCH/RUNNING projection renders recognizably into the string.
- Model reconcile-level tests on `test/reconcile-downgrade-event.test.mjs` / `test/reconcile-next-action.test.mjs`.
- Whole suite green: `for t in test/*.test.mjs; do node "$t"; done`.

## Context

survives every regen because it lives in the ledger, not the mirror — one writer on the mirror, no cache

is the tmp-write+renameSync helper; session-start.mjs:121 reads `p.nextAction` as a string.
