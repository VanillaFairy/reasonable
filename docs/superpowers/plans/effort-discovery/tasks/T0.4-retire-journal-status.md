# T0.4 — Retire the journal's per-work-order `status` field

**Spec:** §5.2 (F8, v2-revised). **Wave:** 0b. **Depends on:** T0.1 (the ledger fold must be the source
before the journal field can go). **Model:** standard.

## Objective

`journal.workOrders[id].status` is a **lossy duplicate** of the T0.1 ledger fold, and it drifts: reconcile
already lands its corrections as `node-downgraded` ledger events, but the journal `status` (fenced to
`journal-writer`) never gets the correction — three disagreeing values. The v2 fix is to **retire the
field**, not to add persist-back choreography. A field that does not exist cannot lie.

`journal.workOrders` conflates two data classes; split them:
- **Lane registry** — `worktree`, `branch`, `commits`, `mergedCommits?` (and existing pointer fields). No
  ledger twin; read by `commit-accounting.mjs` and reconcile's orphan-worktree accounting. **KEEP.**
- **`status`** — the lossy duplicate. **REMOVE.**

## Scope

- **Edit:** `agents/journal-writer.md` (stop instructing a per-WO `status` write; keep the lane registry +
  write-ahead lane registration + program-counter pointers `currentVerticalSlice`/`phase`).
- **Edit:** `lib/reconcile.mjs` (any remaining `status` *reads* that T0.1 did not already convert →
  consume the fold; a legacy `status` still present is cross-checked + warned, never trusted).
- **Verify (do not edit unless it reads status — it does not):** `lib/commit-accounting.mjs` stays correct
  (it reads only `commits`/`mergedCommits`).
- **Do NOT edit:** `wo-status.mjs` (done in T0.1), `ledger.mjs`, `docs/*`.

## Interface (from shared/interfaces.md §T0.4)

- `journal.workOrders[id]` = lane-registry fields only, **no `status`**.
- `journal-writer` keeps write-ahead lane registration but no longer stamps `status`.
- Every status reader uses `foldWorkOrderStatuses()`; legacy `status` → warn-on-mismatch, never governs.

## Positive constraints (DO)

- In `agents/journal-writer.md`, remove the instruction that the scribe records `pending | dispatched |
  checkpointed | merged | dead-end` status transitions on `workOrders`, and reword the write-ahead section
  so it still **registers the lane** (worktree/branch) ahead of dispatch and bumps `dispatchEpoch`, but does
  not write `status`. Keep every other responsibility (inbox, cost block, `currentVerticalSlice`, `phase`).
- Audit `reconcile.mjs` for `wo.status` reads that survived T0.1. Convert them to the fold. If reconcile's
  crash-recovery loop keyed on `wo.status === 'dispatched'|'checkpointed'`, key it on the fold's
  `running`/checkpoint state instead (the git-aware downgrade logic stays; only its *input* moves from
  journal to fold).
- Where a legacy journal carries `status`, emit the same cross-check note shape T0.1 established.

## Negative constraints (DO NOT)

- Do NOT remove or rename any lane-registry field (`worktree`/`branch`/`commits`/`mergedCommits`). Breaking
  `commit-accounting.mjs` is a hard fail.
- Do NOT change the ledger, the fold, or `progress`/tree status (those are separate vocabularies).
- Do NOT weaken the journal-writer's single-writer / no-SHA-forgery / inbox rules.
- Do NOT introduce a new field to replace `status`.

## Tests (test-first)

- A reconcile test where `journal.workOrders` has **no** `status` at all (only lane registry) → reconcile
  derives correct statuses from the ledger fold (proves the field is truly unneeded).
- A reconcile test where a legacy `status` **disagrees** with the fold → fold wins + a warning note.
- A `commit-accounting` test (or reuse an existing one) proving it still partitions commits with a
  status-free journal.
- Confirm the journal-writer constitution change doesn't break any test that parses journal shape.

## Acceptance

- Whole suite green with journals that carry **no** per-WO `status`.
- `journal-writer.md` no longer instructs a `status` write; lane registry + pointers intact.
- `commit-accounting.mjs` verified correct against a status-free journal.
