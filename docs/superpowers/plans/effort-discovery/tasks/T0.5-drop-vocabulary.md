# T0.5 — Drop/blocking vocabulary: structured, ledger-borne, fold-closed

**Spec:** §5.6 (F12, v2-revised). **Wave:** 0b. **Depends on:** T0.1 (fold), T0.4 (no journal status → the
ledger is the only place a drop can live). **Model:** standard.

## Objective

Three defects in how a work order is dropped or blocked:
1. `redispatch-guard.mjs` binds on `dead-end`/`verdict`(infeasible). **These ARE emitted** (dead-end
   ceremony; consumed by `dead-ends.mjs`/reconcile) — Defect B — but the guard has **no** binding on the
   ordinary blocking events this pipeline actually produces (`node-failed`, `amendment` drops).
2. An `amendment` drop is recorded only by a hand-deleted WO file — no structured, machine-foldable record.
3. A `node-failed` is "closed" today by nothing precise — reconcile has no closure fold.

Fix: make drops a **structured field on the ledger event**, ADD the missing redispatch bindings (keep the
existing one), and give reconcile a `resolvesSeq` closure fold.

## Interface (from shared/interfaces.md §T0.5)

- **Ledger grammar (additive, optional):** `amendment` and `ratification` events may carry
  `drops: [{ workOrder: string, supersededBy?: string }]` and `resolvesSeq: number`. Optional in
  `EVENT_SCHEMAS` (old events lack them); validate the shape when present.
- **redispatch-guard:** ADD binding on (`node-failed` with a blocking-class `reason`) and (`amendment` with
  `drops[].workOrder === id`). KEEP `dead-end` + `verdict`(infeasible, survivedSkeptic). Exit 2 = blocked,
  0 = clear (unchanged).
- **reconcile closure:** a `node-failed` at seq S is closed only by a later `ratification`/`amendment` with
  `resolvesSeq === S`. Never by coincidental id mention.

## Positive constraints (DO)

- Add `drops`/`resolvesSeq` to `EVENT_SCHEMAS` as **optional**; add a small validator (if `drops` present,
  it is an array of `{workOrder:string, supersededBy?:string}`; if `resolvesSeq` present, a number). Do not
  make them required — that would reject all historical events.
- In `redispatch-guard.mjs`, extend the filter to also match `node-failed` (with a blocking-class `reason`)
  and `amendment` drops for the WO — **in addition to** the existing predicate. Keep exit-code semantics.
- In `reconcile.mjs`, use `resolvesSeq` to decide whether a `node-failed` is still open (feeds the
  blocked/dropped surfacing). Keep it consistent with T0.1's fold semantics — same closure meaning.

## Negative constraints (DO NOT)

- Do NOT remove the existing `dead-end`/`verdict` binding (Defect B — it is real and consumed elsewhere;
  `dead-ends.mjs` keeps the identical `isBindingDeadEnd` predicate in lockstep — do not desync them).
- Do NOT make `drops`/`resolvesSeq` required, or change any other event's schema.
- Do NOT infer a drop from a hand-deleted WO file — the file deletion stays the fence-visible act, but the
  **ledger event** is the truth record the fold keys on.
- Do NOT rewrite `wo-status.mjs`; coordinate closure semantics via a shared exported helper if needed.

## Tests (test-first)

- `test/redispatch-guard.test.mjs`: a WO with a matching binding `dead-end` → exit 2 (regression-guard the
  KEPT path); a WO with a blocking `node-failed` (unresolved) → exit 2; the same WO after a
  `ratification{resolvesSeq}` closes it → exit 0; an `amendment` drop → exit 2; a clean WO → exit 0.
- `ledger.mjs` schema: an `amendment` with well-formed `drops`/`resolvesSeq` validates; a malformed `drops`
  (not an array / missing `workOrder`) is rejected; an `amendment` with neither still validates.
- `reconcile.mjs`: `node-failed` with no matching `resolvesSeq` → surfaced open/blocked; with a matching
  `ratification{resolvesSeq}` → closed.
