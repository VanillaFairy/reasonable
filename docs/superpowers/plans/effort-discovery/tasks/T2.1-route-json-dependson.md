# T2.1 — route.json (new machine artifact) + WO `dependsOn` + writers + loader

**Spec:** §7.2 (inputs — `route.json` for the forward frontier; WO `dependsOn` predicate). **Wave:** 2a
(first Layer-2 wave; file-disjoint from `reconcile.mjs`). **Depends on:** Layer 0 + Layer 1 (merged).
**Model:** standard.

## Objective

Land the two new **structured inputs** the deterministic `nextAction` will consume, plus the code + prose
that produce them — WITHOUT touching `reconcile.mjs` or computing any directive (that is T2.2+). Concretely:
a new `.reasonable/route.json` artifact + its pure loader `lib/route.mjs`, and a new required `dependsOn`
field on the work-order schema wired into the two agents that author work orders.

## Scope
- **Create:** `lib/route.mjs` (pure loader), `test/route.test.mjs` (new).
- **Edit:** `agents/work-order-writer.md` (JSON template gains `dependsOn`), `agents/route-planner.md`

  persists the route (write `route.json` alongside `route.md`). Possibly a WO-spec test fixture if one asserts

- **Do NOT edit:** `lib/reconcile.mjs`, `lib/ledger.mjs`, `lib/progress-map.mjs`, `lib/next-action.mjs` (does
  not exist yet — T2.2 creates it). Do NOT compute readiness or any directive here. Do NOT edit
  `docs/artifacts.md`/`DESIGN.md`/`architecture.md` (T2.doc owns them — report the grammar delta instead).

## Interface (authoritative — from `shared/interfaces.md` §Layer 2 → T2.1)

### `.reasonable/route.json` (new machine artifact, single-writer, SEALED-class — no fence change)
```json
{ "slices": ["walking-skeleton", "expr-eval", "…"],   // ordered vertical-slice ids, best-first; [0] is the walking skeleton
  "ratifiedAt": "2026-07-06T12:00:00+02:00",           // local ISO — when the human ratified this ordering
  "ledgerSeq": 42 }                                     // ledger seq at ratification — back-pointer into the truth log
```
- Carries only the slice **order**. WO→slice membership stays on each WO spec's existing `verticalSlice` field.
- `route.md` is unaffected — it stays human narration, **never parsed**. `route.json` is the machine twin.

### `lib/route.mjs` — the pure loader
```js
export function readRoute(effortRoot)  // -> { slices, ratifiedAt, ledgerSeq } | null
```
- Absent `route.json` → `null` (forward-compat: a pre-route.json effort must brief without a frontier, never
  crash). Present-but-invalid (not an object; `slices` missing or not an array of non-empty strings) → `null`
  **plus a surfaced diagnostic** — be **conservative**: never fabricate or partially-repair an order. Prefer a
  shape like `readRoute(root) -> { route|null, diagnostic?|null }` OR return `null` and expose the diagnostic
  via a second export — pick whichever reads cleanest against how `reconcile`/session-start consume it, and
  pin the exact signature in your report so T2.2 matches it. Node builtins + relative imports only (dep-free).
- Model this module on `lib/wo-status.mjs` (a small, pure, single-purpose parse module). Reuse `readJson` from
  `lib/effort.mjs` for the file read if it surfaces parse failure suitably; else a local guarded `JSON.parse`.

### Work-order schema — new required `dependsOn: [workOrderId]`
- Array of WO-id strings; `[]` means no dependencies. Because a WO spec is **write-once / immutable**
  (`work-order-writer` never patches an existing spec), `dependsOn` MUST be written at spec-creation time.
- `agents/work-order-writer.md`: add `"dependsOn": ["<from route-planner's proposed readiness edges, else []>"]`
  to the JSON template (match the existing template's field ordering/comment style).
- `agents/route-planner.md`: instruct it to **propose** each WO's `dependsOn` — the **readiness/ordering edge**
  (its predecessor's output must exist first). This is **distinct** from the footprint independence edges the
  route-planner already computes for wave packing (locus/contract/resource overlap) — do NOT conflate the two
  (recon gotcha #5): footprint independence = "can two WOs run in the same wave?"; `dependsOn` = "does this
  WO's input even exist yet?". Add the field to its proposal shape and say so in one clear sentence.

### The readiness predicate this task ENABLES (implemented in T2.2 — documented here so `dependsOn` is authored correctly)
A WO is **ready** iff: it is not itself terminal (not `done`/`dropped`/canceled-terminal) AND every id in its
`dependsOn` has status **`done`** AND it is not redispatch-guard-flagged. (Correction C: `done` is
`reconcile()`'s `workOrderStatuses` value — `merged` already folds into it; there is no `green`. Correction D:
a canceled dep is terminal-abandoned and does NOT satisfy `dependsOn`.) T2.1 only makes the field EXIST and be
POPULATED; the predicate itself is T2.2.

## Persisting `route.json` (the writer)
- The route is human-ratified orchestration state. Today the orchestrator persists `route.md` from analysis
  Step 3 (see `skills/analysis/SKILL.md`) and re-sorts it per `route-planner` proposals (which the
  orchestrator ratifies). Add the machine twin: whenever the orchestrator writes/updates `route.md`, it also
  writes `route.json` with the ordered slice ids, `ratifiedAt` (local ISO), and `ledgerSeq` (the current
  ledger seq). Express this as a prose instruction in the same skill/step that owns `route.md` — `route.json`
  is SEALED-class (orchestrator-only) automatically; **no `fence.mjs` change is needed or allowed here.**
- Keep it minimal: do not invent per-slice tier fields or a DAG in `route.json` (YAGNI — tier overrides stay
  in `route.md`/config; the footprint DAG is computed fresh, never stored).
